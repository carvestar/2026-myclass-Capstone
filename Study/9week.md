01 일상에서의 프로세스 의미
1. 프로세스
 1) 일을 처리하는 과정 또는 순서, 공정
 2) 주어진 일을 해결하기 위한 목적
 3) 그 순서가 정해져 수행되는 일련의 절차
2. 프로세스 목적
 1) 이전에 얻은 노하우 전달 -> 시행착오 감소 -> 빠른 적응
 2) guide 역할

02 소프트웨어 프로세스 모델의 이해
1. 소프트웨어 개발 과정
 1) 작은 규모의 소프트웨어 개발 -> 개집 짓기
 2) 대규모의 소프트웨어 개발 과정 -> 빌딩 짓기

2. 소프트웨어 프로세스 모델
 1) 소프트웨어 프로세스 모델 정의
  (1) 소프트웨어 개발 생명 주기 (SDLC)
  (2) SW를 어떻게 개발할 것인가에 대한 전체적인 흐름을 체계화한 개념
  (3) 순차적인 단계로 이루어짐
 2) 소프트웨어 프로세스 모델 목적
  (1) 소프트웨어 개발의 전 과정을 하나의 프로세스로 정의
  (2) 주어진 예산과 자원으로 개발하고 관리하는 방법을 구체적으로 정의
  (3) 고품질의 소프트웨어 제품 생산 -> 목적 (BBB)

03 주먹구구식 모델
1. 주먹구구식 모델
 1) 즉흥적 소프트웨어 개발
 2) 정확한 앞뒤 없이 일을 대충 처리할 때 쓰는 말
 3) 공식적인 가이드 라인이나 프로세스가 없는 개발 방식
 4) 요구 분석 명세서나 설계 단계 없이 간단한 기능만을 정리하여 개발하는 형태
 5) 일단 코드를 작성하여 제품을 만들어본 후 요구 분석, 설계, 유지보수에 대해 생각

2. 주먹구구식 모델 사용 및 단점
 1) 주먹구구식 모델의 사용
  (1) 개발자 한 명이 단시간에 마칠 수 있는 경우에 적합
  (2) 대학 수업의 한 학기용 프로젝트 정도
 2) 주먹구구식 모델의 단점
  (1) 개발 순서나 단계별로 문서화된 산출물 없음 -> 관리 및 유지보수 어려움
  (2) 프로젝트 전체 범위 알 수 없을 뿐더러 좋은 아키텍처를 만들 수 없음
  (3) 일을 효과적으로 나눠 개발할 수 없으며 프로젝트 진척 상황 파악할 수 없음
  (4) 계속적 수정으로 인해 프로그램 구조 나빠져 수정 매우 어려워짐

04 선형 순차적 모델
1. 선형 순차적 모델
 1) Linear sequential 모델
 2) waterfall 모델
 3) Classic life cycle

2. 폭포수 모델 장점
 1) 관리 용이
 2) 체계적인 문서화
 3) 요구사항의 변화가 적은 프로젝트에 적합

3. 폭포수 모델 단점
 1) 앞 단계가 완료되어야 단계 수행 가능
 2) 각 단계의 결과물이 완벽한 수준으로 작성되어야 다음 단계에 오류를 넘기지 않음
 3) 사용자가 중간에 가시적인 결과를 볼 수 없음

7. 프로토타입 모델 개발 절차
 1) 요구사항 정의 및 분석
 2) 프로토타입 설계
  (1) 완전한 설게 대신 사용자와 대화할 수 있는 수준의 설계
  (2) 입출력 화면을 통한 사용자 인터페이스 중심 설계
 3) 프로토타입 개발
  (1) 완전히 동작하는 완제품 개발이 아님
  (2) 입력 화면을 통한 사용자의 요구 항목 확인
  (3) 출력 결과를 통해 사용자가 원하는 것인지 확인

07 통합 프로세스 모델 (진화적 프로세스 모델)
1. 통합 프로세스 모델 (Unified Process)
 1) 폭포수 모델 문제점(backward가 안 됨) -> 해결 방안 -> 반복적 개발 방법
 2) R(요구 사항 정의), A(분석), D(설계), C(구현), T(테스트), M(유지보수)
 3) 통합 프로세스 모델 그림 - 기말 나옴 그림 중요 -
     (9주차 ppt p.28 그림 참고)
  (1) 단계: 도입, 구체화, 구축, 전이
  (2) 전이: 실제 사용자에게 전달
  (3) 반복: 필요 시 반복
  (4) 세로축: 각 단계를 어떻게 할 건
  (5) 하늘색 면적 그래프: 각각 단계에 어떤 영역의 일을 많이 해야 하는지
    * 도입 단계 때는 비즈니스 모델링과 요구 사항 정의를 많이 해야 함
  (6) 형상 관리 ~ 환결 점검: 위 영역들과 별개
    * 형상 관리 = 문서화
    * 환경 점검 = 개발/실실적업무 환경

2. UP 모델에서 비즈니스 모델링
 1) 소프트웨어 개발에 앞서 시스템이 운영될 비즈니스 환경을 이해하고 분석하는 핵심 작업
  (1) 프로젝트 초기에 가장 집중적으로 이루어짐
  (2) 프로젝트 전반에 걸쳐 비즈니스 맥락을 유지하기 위해 지속적으로 영향
 2) 비즈니스 모델링 주요 역할과 특징
  (1) 시스템 목적 및 범위 정의
    * 소프트웨어가 해결해야 할 비즈니스 문제 명확히 파악
    * 전체 시스템의 맥락 이해 -> 프로젝트 왜 필요한지, 어떤 비즈니스 가치 창출할지 정의
  (2) 현실 세계의 개념화
    * 실제 업무에 사용하는 개념을 시각하여 Domain Model 구축
    * 소프트웨어 구조가 아닌 '업무 개념 중심'의 모델
  (3) 업무 프로세스 분석
    * 조직 내 수행되는 업무 절차를 흐름도나 BPMN 등으로 시각화
    * 어떤 역할을 하는 주체가 어떤 작업을 수행하는지 분석
  (4) 프로젝트 기반 제공
    * 도출된 비즈니스 규칙과 개념은 이후의 '요구 사항 정의' 및 '분석', '설계' 단계의 초석
    * Ex) 도메인 모델에서 정의한 용어 개념이 시스템의 클래스 구조를 설계하는데 직접적을 활용

3. Business Process Model and Notation(BPMN) (BBBBB)
 1) 조직 내의 업무 흐름을 시각화하고 표준화하기 위한 글로벌 그래픽 표기법
     (9주차 ppt p.30 그림 참고)

4. 통합 프로세스(UP) 빙법
     (9주차 ppt p.31 그림 참고)
 1) 비즈니스 모델링, 프로젝트 계획, 요구 사항 정의, 분석 및 설계, 구현, 테스트, 배치 / 평가
 2) 사이클

5. 통합 프로세스 모델 절차
 1) 도입 2) 구체화 3) 구축 4) 전이 5) 공통 작업

08 애자일 프로세스 모델 - 기말 나옴 -
1. 애자일 프로세스 모델
 1) ahile = 날렵한, 민첩한
 2) 애자일 프로세스 모델: 고객 요구에 민첩하게 대응하고 그때그때 주어지는 문제를 풀어나가는 방법론
 3) 애자일의 기본 가치 (애자일 선언문)
  (1) 프로세스와 도구 중심이 아닌 개개인과의 상호 소통 중시
  (2) 문서 중심 아닌 실행 가능한 소프트웨어 중시
  (3) 계약과 협상 중심 아닌 고객과의 협력 중시
  94) 계획 중심 아닌 변화에 대한 민첩한 대응 중시

2. 애자일 프로세스 모델 이해
 1) 애자일 개발 방법
  (1) 반복적인 개발을 통한 잦은 출시 -> 목표
     프로토타입 개발 -> 사용자 확인 -> 일부 기능 사용

3. 애자일 개발 방법론 (스크럼)
 1) 스크럼 개발 프로세스
  (1) 소프트웨어 개발보다 팀의 개선과 프로젝트 관리를 위한 애자일 방법론
  (2) 경험적 관리 기법
  (3) 구체적인 프로세스 명확하게 제시하지 않음
  (4) 개발 팀을 운영하는 효율적인 운영 방식
 2) 팀원들이 럭비의 스크럼처럼 똘똘 뭉쳐서 짧은 구간을 전력 질주하며 목표 달성 (BBBB)
  제품 기능 목록 작성 -> 스프린트 계획 회의 -> 스프린트 수행 -> 스프린트 개발 완료 (최종 제품) -> 스프린트 완료
